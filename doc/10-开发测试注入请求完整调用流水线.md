# 10 · 开发测试「发送一次」完整调用流水线

> 场景：打开 `首页 / 遥控 / 开发测试`，点一次「发送」（或模拟里发一帧）。  
> 对照日志片段：`POST /dev-api/payload/telemetry/dev/can-yc` + 一串 `sys_user/sys_dept/sys_role/...` SQL + `ROLLBACK` + 注入成功。
>
> 结论先说：
> 1. **那一串 SQL 不是业务在读遥测库，而是若依框架每次鉴权都要从 SQLite 把当前用户“拼完整”**。
> 2. **注入接口本身几乎不碰 SQLite**，核心动作是：校验 HEX → 解析遥测 → 写 Redis。
> 3. SQLAlchemy 的 `BEGIN` / `ROLLBACK` 是只读鉴权会话的正常收尾，不是业务回滚失败。

---

## 一、日志为什么这么“长”

一次请求里，框架日志通常会看到两份同样的 SQL（`sqlalchemy.engine.Engine` + 项目 `log_util` 的 `callHandlers`），所以肉眼感觉“膨胀了一倍”。  
真正执行的查询大约只有这 5 条（与日志一致）：

| # | SQL 表 | 目的 |
| - | ------ | ---- |
| 1 | `sys_user` | 按 `user_id` 取用户基本信息 |
| 2 | `sys_user` ⋈ `sys_dept` | 取部门 |
| 3 | `sys_user` ⋈ `sys_user_role` ⋈ `sys_role` | 取角色 |
| 4 | `sys_user` ⋈ `sys_user_post` ⋈ `sys_post` | 取岗位 |
| 5 | `sys_menu`（超级管理员）或经角色关联的菜单 | 取权限/菜单，拼 `permissions` |

这些都发生在进入业务函数 **之前**。业务成功后再有：

```text
INFO ... inject_can_yc_test - 注入CAN遥测测试数据成功 device=can:0:0:0 type=FF
INFO: 127.0.0.1:... - "POST /dev-api/payload/telemetry/dev/can-yc HTTP/1.1" 200 OK
```

以及会话结束时的 `ROLLBACK`（只读事务，无写库）。

## 二、端到端流水线（按时间顺序）

```text
[前端] 开发测试页
   │  点击「发送」 / 模拟定时发送
   ▼
injectCanYcTest({ deviceId, hex })
   │  axios：带 Authorization: Bearer <token>
   │  URL：POST /dev-api/payload/telemetry/dev/can-yc
   ▼
[网关/Vite 代理] /dev-api → 后端根路径
   ▼
[FastAPI 依赖链]
   ① get_db()                → 打开 AsyncSession（SQLite）
   ② PreAuthDependency       → 登录校验 + 组装 CurrentUser
   ③ UserInterfaceAuth(...)  → 检查 permissions 含 payload:devtest:view
   ▼
[业务] inject_can_yc_test()
   └─ PayloadTelemetryService.inject_can_yc(redis, deviceId, hex)
         ├─ 校验帧 / 解析字段（纯内存 + TeleMetry 配置）
         ├─ Redis SET 最新帧
         └─ Redis ZADD 全字段曲线点
   ▼
[收尾] AsyncSession 退出 → ROLLBACK（只读）→ 200 OK
```

## 三、前端：点一次「发送」走了哪些函数

文件：`ruoyi-fastapi-frontend/src/views/payload/telecontrol/devtest/index.vue`

| 步骤 | 函数 / 动作 | 做什么 |
| ---- | ----------- | ------ |
| 1 | `handleSend()` / 模拟里调 `sendFrame()` | 从文本框或内存 `frameBytes` 取帧 |
| 2 | `parseHex` / `formatHex` | HEX 字符串 ↔ 字节 |
| 3 | （模拟时）`incrementSimFields` + `recalcChecksum` | 改字段并重算校验和 |
| 4 | `sendFrame(bytes)` | 调 API |
| 5 | `injectCanYcTest`（`api/payload/telemetry.js`） | `POST /payload/telemetry/dev/can-yc` |
| 6 | axios 拦截器（`utils/request.js`） | 自动加 `Authorization`；可选防重复提交 |

按钮上还有 `v-hasPermi="['payload:devtest:view']"`：这是**前端菜单/按钮级权限**，和后端接口鉴权是两层；前端过了不等于后端一定过，反过来也一样。

请求体形如：

```json
{ "deviceId": "can:0:0:0", "hex": "00 BF 3A FF ..." }
```

## 四、后端依赖链：为什么会有一堆 SQL

路由定义（节选）：

```python
# module_payload/controller/payload_telemetry_controller.py
payload_telemetry_controller = APIRouterPro(
    ..., dependencies=[PreAuthDependency()]          # 整组路由：先登录
)

@...post('/dev/can-yc', dependencies=[UserInterfaceAuthDependency('payload:devtest:view')])
async def inject_can_yc_test(...):
    ...
```

### 4.1 `get_db()` — 打开 SQLite 会话

文件：`config/get_db.py`

- `async with AsyncSessionLocal() as current_db: yield current_db`
- 日志里的 `BEGIN (implicit)` 就是这个会话开始
- 请求结束会话关闭：本次**只读**，没有 `commit`，所以日志是 `ROLLBACK`（正常收尾，不是失败）

### 4.2 `PreAuth.__call__` — 必须登录

文件：`common/aspect/pre_auth.py`

1. 看路由是否在排除名单（本接口不在）
2. 读请求头 `Authorization`
3. 空 → `AuthException('用户未登录…')`
4. 有 → `LoginService.get_current_user(request, token, db)`

### 4.3 `LoginService.get_current_user` — 拼当前用户（SQL 主力）

文件：`module_admin/service/login_service.py`

| 顺序 | 动作 | 存储 |
| ---- | ---- | ---- |
| A | `jwt.decode(token, …)` 取 `user_id` / `session_id` | 内存 |
| B | `UserDao.get_user_by_id(db, user_id)` | **SQLite 五次查询**（见下一节） |
| C | Redis `GET` access_token，与请求 token 比对是否仍有效 | Redis |
| D | Redis `SET` 续期 access_token TTL | Redis |
| E | Redis 读系统配置（是否强制改初始密码 / 密码过期天数） | Redis |
| F | 组装 `CurrentUserModel`，角色含管理员则 `permissions=['*:*:*']` | 内存 |
| G | `RequestContext.set_current_user(...)` | 请求上下文 |

### 4.4 `UserDao.get_user_by_id` — 逐条对应你贴的 SQL

文件：`module_admin/dao/user_dao.py`

| 日志 SQL | 函数内变量 | 用途 |
| -------- | ---------- | ---- |
| `SELECT … FROM sys_user WHERE … user_id=?` | `user_basic_info` | 用户是否存在/启用 |
| `sys_user JOIN sys_dept …` | `user_dept_info` | 部门对象挂到 `user.dept` |
| `sys_user … JOIN sys_role …` | `user_role_info` | 角色列表；`role_id==1` 视为超管 |
| `sys_user … JOIN sys_post …` | `user_post_info` | 岗位 |
| `SELECT … FROM sys_menu WHERE status='0'` | `user_menu_info` | 超管：拉全部菜单；非超管：按角色过滤后取 `perms` |

> 你的日志里菜单查询是「全表 `sys_menu`」，说明当前用户角色里带有管理员角色（`role_id=1`），权限直接变成 `*:*:*`，后面接口权限检查会立刻通过。

### 4.5 `CheckUserInterfaceAuth('payload:devtest:view')` — 接口权限

文件：`common/aspect/interface_auth.py`

- 从 `RequestContext.get_current_user()` 取 `permissions`
- 含 `*:*:*` 或精确匹配 `payload:devtest:view` → 通过
- 否则 `PermissionException('该用户无此接口权限')`
- **这一步不再查库**，只做内存判断

到这里，你贴的 SQL / ROLLBACK 已经**全部解释完**；下面才是「开发测试」真正干活的部分。

## 五、业务主体：`inject_can_yc`（不写 SQLite）

文件：

- Controller：`payload_telemetry_controller.inject_can_yc_test`
- Service：`PayloadTelemetryService.inject_can_yc`
- 帧工具：`module_payload/cfg/can_yc_frame.py`
- Redis：`module_payload/redis_store.py`：`set_telemetry` / `append_curve_points`

### 5.1 流水线明细

| # | 函数 | 作用 | SQLite | Redis | 其他 |
| - | ---- | ---- | ------ | ----- | ---- |
| 1 | `inject_can_yc_test` | 接 body，调 service，打业务日志 | 否 | 否 | `logger.info` 成功日志 |
| 2 | `hex_to_bytes` | HEX → bytes | 否 | 否 | |
| 3 | `verify_can_yc_frame` | 长度、校验和、帧类型 `0x3A` | 否 | 否 | 失败抛业务异常 |
| 4 | `parse_can_yc_frame` | 抽出 `dataType`（如 `FF`）、payload | 否 | 否 | |
| 5 | `TeleMetryCfgManager.parse_hex` | 按配置把 payload 解成若干字段行 | 否 | 否 | 读内存中的 `TeleMetryCfg.json` |
| 6 | 组装 `fields[{id,name,value,show,hex,unit}]` | 供表页/曲线使用 | 否 | 否 | |
| 7 | `set_telemetry` | 覆盖写最新一帧 | 否 | **SET** `payload:{device}:tm:{TYPE}`、`…:ts` | |
| 8 | `append_curve_points` | 每个可数值字段追加一点 | 否 | **Pipeline：ZADD + ZREMRANGEBYRANK**（每字段） | 上限 `CURVE_MAX_POINTS=50000` |
| 9 | `ResponseUtil.success` | 返回 `{deviceId,dataType,fieldCount,ts,…}` | 否 | 否 | HTTP 200 |

### 5.2 Redis 写了什么（一次发送）

以 `deviceId=can:0:0:0`、`dataType=FF`、约 N 个数值字段为例：

```text
SET  payload:can:0:0:0:tm:FF          # JSON：最新整表
SET  payload:can:0:0:0:tm:FF:ts       # 时间字符串
ZADD payload:can:0:0:0:curve:FF:JGB001 { "ts|val": ts }   # × N 字段
ZREM… 裁剪每个 ZSet 只留最近 5 万点
```

**不写**：SQLite 业务表、指令历史、曲线订阅集合（订阅已移除）。

### 5.3 和鉴权 Redis 的区别

| Key 用途 | 示例 | 谁用 |
| -------- | ---- | ---- |
| 登录 token | `access_token:{session_id}` | `get_current_user` |
| 系统参数 | `sys_config:sys.account.…` | 密码策略判断 |
| 地检遥测 | `payload:…:tm:…` / `payload:…:curve:…` | `inject_can_yc` |

同一次请求里 Redis 两边都会碰到，但语义完全不同。

## 六、对照你贴的日志时间轴（~08:58:35）

| 时间点（同秒内） | 含义 |
| ---------------- | ---- |
| `BEGIN` | `get_db` 开事务 |
| 5 组 SELECT（user/dept/role/post/menu）各打两遍日志 | `UserDao.get_user_by_id` 鉴权组装 |
| （日志未打出）Redis GET/SET token | token 校验与续期 |
| （业务无 SQL）`inject_can_yc` 写 Redis | 真正注入 |
| `inject_can_yc_test … 注入…成功 type=FF` | Controller 业务日志 |
| `POST …/dev/can-yc 200` | Uvicorn access log |
| `ROLLBACK` | 鉴权用 AsyncSession 正常关闭 |

所以：**日志很吵，不等于注入逻辑在扫库。**

## 七、判断清单（这次请求后台“判了什么”）

按失败优先级：

1. **有没有 Token** → 未登录  
2. **JWT 能不能解 / user_id 有没有** → token 非法  
3. **SQLite 里用户是否存在且未删/启用** → token 非法  
4. **Redis 里 token 是否仍是当前有效会话** → 请重新登录  
5. **权限列表是否含 `payload:devtest:view`（或 `*:*:*`）** → 无接口权限  
6. **HEX / 帧长 / 校验和 / 帧类型** → 业务 `ServiceException`  
7. **`parse_hex` 是否解出字段** → 解析无结果  

全部通过才会写 Redis 并 200。

## 九、开发提示

- 想少看 SQL 日志：调低 SQLAlchemy `echo` / 引擎日志级别（与业务无关）。  
- 每秒「开始模拟」会每秒重复整条鉴权 SQL 流水线——这是当前若依设计，不是注入函数重复解析用户 5 次以外的“额外逻辑”。  
- 若将来要减负，方向是：**用户信息/权限缓存到 Redis**，而不是改 `inject_can_yc`。
