| 字段 | 说明 |
|------|------|
| ID | 流水号，如 `OPEN-001` |
| 模块 | 如组装器 / 遥测解析 / Redis / 前端 |

| 优先级 | `高` / `中` / `低` |

## 开放中

### OPEN-001 · 工程遥测组装器：未完成拼装无超时清理

| 项 | 内容 |
|----|------|
| 状态 | 开放 |
| 优先级 | 中 |
| 模块 | 组装器 `eng_tm_subpkt` |
| 登记日 | 2026-07-22 |
| 来源 | 序号组帧丢弃逻辑评审；`module_payload/assemblers/eng_tm_subpkt.py` |

**现状**

- 多包子包按序号连续拼装；遇不连续 / 非首帧 / 单包冲掉等会主动丢缓存。
- 若已收到部分子包（如仅有 `1/N`）后长时间不再来包，**缓存会一直保留**，直到：
  - 再来首帧 / 单包；
  - 序号不连续；
  - 会话参数（源/目的/总包数）变化；
  - 组装器 `reset()`（换绑定或进程重启等）。

**缺口 / 为何未做**

- 缺少「未完成拼装超时自动清空」；参考图像多帧组装器（`test/StrictImageAssembler.cpp`）通常有超时，工程遥测侧尚未对齐。
- 长时间残缺缓存可能占内存，并影响后续同会话误拼（虽有连续序号约束，仍不如超时干净）。

**建议方案（可选）**

- 在 `EngTmSubpktAssembler` 记录 `_assembly_deadline`（或每槽/首包到达时间）。
- `feed()` 入口检查：若已超时则 `_drop_assembly('未完成拼装超时')`，错误进 `payload:error:assembler`。
- 超时时间建议可配置（常量或会话参数），初值可先取 **3s～10s**（待与协议/联调确认）。

**验收要点（可选）**

- 收到 `1/3` 后超过阈值未再收包 → 缓存清空，并有 assembler 错误记录。
- 超时前按序收齐 → 仍正常产出，不误杀。
